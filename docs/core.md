# Core Architecture Guide

このディレクトリは、API・DB・Lambda・SeaORM などの外部技術に依存しない、アプリケーションの中核ロジックを配置するための領域です。

## 目的

- DB が無くても単体テストできる状態を維持する
- Lambda / API Gateway 以外からも同じ処理を呼び出せるようにする
- 業務ロジックを SeaORM、SQL、HTTP、AWS Lambda などの実装詳細から分離する
- Repository trait によって依存性逆転を行い、外側の実装を差し替え可能にする

提示された設計ルールに基づき、新しくチームに参入したエンジニアでも一目で全体像と実装ルールが理解できる、人間向けの /core/README.md を作成しました。
先ほど指摘した「Usecase層からの common への依存許可」や「マッピングの実装場所」の補足も反映し、Rust特有の実装パターン（NewtypeやTrait）を意識した実践的なドキュメントに仕上げています。

# Core Module (DDD & Clean Architecture)

本システムは、**ドメイン駆動設計（DDD）**および**クリーンアーキテクチャ**の原則に基づき、ビジネスロジックを技術的詳細（データベースや外部API、サーバーレス環境など）から完全に分離した疎結合な設計を採用しています。

開発時は、以下の共通設計原則と各レイヤーのルールを必ず遵守してください。

---

## 🛡️ 1. 共通設計原則（アーキテクチャ・ガードレール）

### 🔄 依存関係の方向（厳格に強制）
依存関係の矢印は、常に **外側から内側へ** のみ向かいます：
`infrastructure` ──► `usecase` ──► `domain`

上位・外側のレイヤー（例: インフラ層）は下位・内側のレイヤー（例: ドメイン層）に依存できますが、**内側のレイヤーが外側のレイヤーからコードをインポートすることは厳禁**です。

### 🦀 Rustの型安全性の最大化
ドメイン層において、生のプリミティブ型（`String`, `i32` など）をそのまま扱ってはいけません。
必ず**「不変な型（Newtypeパターン）」**として定義し、そのコンストラクタ（例: `fn new()`）の内部でバリデーションを強制してください。これにより、不正な状態のデータがドメイン層に侵入することをコンパイルレベルで防ぎます。

### 🚫 フレームワークおよび外部ライブラリ依存の排除
`domain` 層および `usecase` 層は、特定のウェブフレームワーク（AWS Lambdaなど）やORM（SeaORMなど）に関する知識を**一切持ってはなりません**。
技術的な詳細が変わっても、コアとなるビジネスロジックは無傷で再利用できるように保ちます。

---

## 🏗️ 2. 全体構造と依存関係図

### 2.1. Domain Layer (`/core/domain/`)
システムの最内方に位置し、最も純粋なビジネスルール（ドメイン知識）をカプセル化するレイヤーです。

*   **🚫 依存制限**:
    *   `usecase`, `infrastructure`, `/db/sea_orm_entities`, `/api/lambda` のコードやクレートをインポートすることは**絶対禁止**です。
    *   外部クレート（ライブラリ）への依存は原則禁止です（`serde` や時間操作ライブラリなど、標準的かつ汎用的なツールのみ例外的に許可）。
*   **💡 実装内容**:
    *   **Value Objects (Newtypes)**: 不変（Immutable）な構造体。生成時に必ずバリデーションを行い、不正なデータがシステム内に存在できないようにします。
    *   **Entities**: IDによって識別され、状態が変化していくドメインオブジェクト。
    *   **ドメインルール（振る舞い）**: データ検証や状態遷移のロジックは、構造体のメソッド（`pub fn`）としてカプセル化します（データだけの「貧血症モデル」を避ける）。
    *   **Repository Trait (インターフェース)**: 永続化（保存・取得）を抽象化するインターフェース。`async-trait`等を用いて定義します。

---

### 2.2. Usecase Layer (`/core/usecase/`)
アプリケーションの実行手順（ユースケースやシナリオ）を制御するレイヤーです。

*   **🚫 依存制限**:
    *   `domain` および `common` 層のみインポート可能です。
    *   `infrastructure` や `/db/sea_orm_entities`, `/api/lambda` をインポートしてはなりません。
*   **💡 実装内容**:
    *   **Use Cases (Interactors)**: 入力（DTO）を受け取り、Domain層のRepository Traitを介してエンティティを取得・操作し、結果を返す一連の手順を記述します。
    *   **Input/Output DTO**: 外部（Lambda等）とデータをやり取りするための、純粋なデータ構造体。
    *   **⚠️ 注意点**: ビジネスルールそのものをこのレイヤーに書いてはいけません。ユースケース層は「ドメインオブジェクトのメソッドを呼び出す手順」の調整に徹してください。

---

### 2.3. Infrastructure Layer (`/core/infrastructure/`)
データベースや外部APIなど、外部技術との具体的な通信を実装する最外方のレイヤーです。

*   **🚫 依存制限**:
    *   `domain`, `usecase`, `common` および、外部の `/db/sea_orm_entities` をインポートできます。
*   **💡 実装内容**:
    *   **具体的なRepository実装**: Domain層で定義されたRepository Traitを、`SeaORM`等を用いて具体的に実装します。
    *   **データモデル変換 (Mapping)**: データベースモデル（`sea_orm_entities`）とDomain層のオブジェクト（Entities/Value Objects）を相互変換するロジックを実装します（Rustの `From` / `Into` トレイトをこのレイヤー内に実装してください）。

---

### 2.4. Common Layer (`/core/common/`)
プロジェクト全体で横断的に使用される、汎用的なユーティリティを配置します。

*   **⚠️ 注意点**: ビジネスロジック（ドメイン知識）は**絶対にここに書かないでください**。
*   **💡 実装内容**:
    *   純粋な「技術ヘルパー」のみを配置します（例: 共通エラー型、ロガーの設定、共通のシステム日時取得ツールなど）。

---

## 📝 3. ドキュメントとコメント（Cargo Doc標準）

すべての構造体、トレイト、列挙型（enum）、および公開（`pub`）関数は、`cargo doc` を通じて正しくレンダリングされるよう、Rust標準のドキュメントコメント（`///` または `//!`）を必ず記述しなければなりません。

*   **🔍 「何（WHAT）」ではなく「なぜ（WHY）」を書く**:
    コードを見ればわかる処理内容（WHAT）を説明する必要はありません。なぜその設計方針にしたのか、なぜそのビジネス制約やロジックが選ばれたのかという**「理由・根拠（WHY）」**に焦点を当てて詳しく記述してください。
*   **🌐 開発者のためのコンテキスト（文脈）の提供**:
    エッジケース、ビジネス上の背景（法規制や業務フローなど）、あるいは外部システムとの連携上の注意点などを盛り込んでください。生成された `cargo doc` が、開発者にとって**「生きた知識ベース（Living Knowledge Base）」**となるようにします。

---

## 🛠️ 開発者向け・レビュー者向けのクイックチェックリスト

コードを書くとき、またはコードレビュー（PR）時は以下をチェックしてください：

- [ ] ドメインモデル（`domain/`）の構造体に、`sea_orm` などのアトリビュート（`#[sea_orm(...)]`）が混入していないか？
- [ ] ユースケース（`usecase/`）の中で、`if user.status == "active"` のようなビジネスルールの条件分岐を直接書いていないか？（ドメインエンティティのメソッドにするべき）
- [ ] データベースへの保存・取得ロジックは、すべて `infrastructure/` のリポジトリ実装の中に閉じ込められているか？
- [ ] 新しい外部ライブラリを追加する際、それが `domain/` に影響を与えていないか？
